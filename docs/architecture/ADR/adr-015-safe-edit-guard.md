# ADR-015: Safe-Edit Guard - PreToolUse Hook, Skills, and Layered Defense

| Field | Value |
|---|---|
| **Status** | Accepted (v0.8) |
| **Date** | 2026-05-31 |
| **Deciders** | Alper Hankendi |
| **Decision** | Ship a PreToolUse hook (`ctxo gate-hook`) that blocks edits to high-impact symbols until the agent has called `get_blast_radius`, plus three model-invoked skills (`ctxo-understand`, `ctxo-safe-edit`, `ctxo-review-pr`) installed by `ctxo init`. Claude Code gets full hook enforcement; Cursor gets skills and rules only. |
| **Relates to** | Epic 7, [ADR-012](adr-012-plugin-architecture-and-monorepo.md) |

---

## Context

The fundamental problem: a permanent solution to "the agent forgets" cannot rely on the agent remembering.

Ctxo exposes 14 MCP tools. `get_blast_radius` is the most critical - it shows what breaks before the agent touches anything. But an agent that has access to the tool can still skip it. Rules and prompts reduce skip rates; they do not eliminate them. The agent reads the rule file at session start, navigates to a target file, spots the function, and edits it - skipping the blast radius check because the local context made the edit seem small.

The failure mode is invisible. There is no error. The edit lands. Dependents break. The agent, not having checked, does not know to fix them.

### Why rules alone are not enough

Rules are guidance. They work when the agent is oriented and unhurried. They fail when the agent is mid-task, context is deep, and the edit feels routine. Three observations from real sessions:

1. Agents follow rules at task start. They stop following them mid-task.
2. The symbols most likely to be skipped are the ones that look small but are high-dependency (utility functions, interfaces, base classes).
3. No rule can interpose between the agent's decision and the Edit call.

### Cursor's limitation

Cursor (through v3.6) has no blocking pre-edit hook mechanism. It has rules and prompts but no equivalent of Claude Code's `PreToolUse` hook. Any gate on Cursor is advisory. This is a platform constraint, not a design choice, and it is documented honestly in the platform matrix.

## Decision

### Layered defense

Three layers work together, each compensating for the one above:

| Layer | Mechanism | Enforces? | Platforms |
|---|---|---|---|
| Rules | AI rule files generated by `ctxo init` | No | All |
| Skills | Model-invoked markdown in `.claude/skills/` and `.cursor/rules/` | No | Claude Code, Cursor |
| Hook | `ctxo gate-hook` registered as PreToolUse in `.claude/settings.json` | Yes | Claude Code only |

Rules reach every platform. Skills provide structured, named instructions the model can reference by name (invoke skill `ctxo-safe-edit`). The hook is the only layer that actually blocks.

### Gate decision

A symbol is flagged as high-impact if it clears both thresholds:

- **Blast radius**: confirmed + likely dependents >= `minDependents` floor
- **PageRank percentile**: symbol falls in the top N% of the codebase by centrality

Both checks must pass. A popular-but-isolated symbol (many importers, no real call chain) does not trigger the gate. An isolated god node with deep calls but no imports does not trigger the gate. Both signals are required to reduce noise.

The thresholds are repo-relative, not fixed numbers. This keeps the guard calibrated across projects of different sizes.

Sensitivity levels:

| Level | PageRank percentile | Min dependents floor |
|---|---|---|
| `strict` | top 30% | 2 |
| `balanced` (default) | top 15% | 3 |
| `lenient` | top 5% | 5 |

### Hook behavior

- **Fail-open**: if `ctxo gate-hook` errors, crashes, or times out, the edit proceeds. The guard is a safety net, not a hard dependency. An unreachable index does not block work.
- **Block-once per session**: once an agent has called `get_blast_radius` for a symbol in a session, subsequent edits to that symbol pass through immediately. No repeated interruptions for the same symbol.
- **Zero false positives for small symbols**: symbols below the thresholds are never blocked. The majority of edits in any session go through without the hook firing.

### Skills

Three skills installed to `.claude/skills/<name>/SKILL.md` (Claude Code) and `.cursor/rules/<name>.mdc` (Cursor):

- `ctxo-understand` - orient with `get_context_for_task` before reading source files
- `ctxo-safe-edit` - run `get_blast_radius` and `get_why_context` before any edit
- `ctxo-review-pr` - run `get_pr_impact` before reviewing a diff

Skills are passive markdown. They do not execute code. Their value is naming: the agent can invoke a skill by name rather than re-deriving the tool sequence each time. The hook is what enforces.

### New CLI commands

- `ctxo blast-radius <symbolId> --json` - compute blast radius for one symbol, JSON to stdout. For CI scripts, jq pipelines, and pre-merge checks.
- `ctxo gate --preview [--json]` - show which symbols the current sensitivity would flag. For tuning before committing to a sensitivity level.

## Consequences

### Positive

- Edits to high-impact symbols are provably preceded by a blast radius check on Claude Code.
- The check fires exactly once per symbol per session - low friction for compliant agents.
- Skills give Cursor users structured guidance even without enforcement.
- `ctxo gate --preview` makes sensitivity tuning observable before it takes effect.
- Fail-open design means the guard never blocks legitimate work if the index is unavailable.

### Negative / Accepted trade-offs

- Claude Code only for enforcement. Cursor and other platforms get skills and rules only. This is a platform constraint with no workaround until those platforms ship pre-edit hooks.
- The guard does not cover renames, deletes, or batch edits via non-Edit tools unless those are also routed through the PreToolUse hook. Scope is Edit calls.
- Session state (which symbols have been checked) is in-process memory. If the MCP server restarts mid-session, previously checked symbols would re-trigger once.

### Deferred

- **Layer-3 UserPromptSubmit nudge** - a prompt-level reminder when the guard has been active. Deferred: hook enforcement already covers the case; nudges risk prompt noise.
- **Cursor reactive stop-hook gate** - blocked on Cursor shipping a pre-edit extension point. Revisit when/if that changes.
- **Tier-3 platform enforcement** - Windsurf, Copilot, and others. Same dependency on platform hooks.
- **Adoption telemetry** - track how often the gate fires, is satisfied, and is bypassed. Gate on confirmed user demand.

## Alternatives considered

### Rules only

Guidance in `CLAUDE.md` and `.cursor/rules/ctxo.mdc`. Already implemented. Insufficient alone - see Context above.

### Periodic reminder hooks

Use `PostToolUse` or session-start hooks to nudge the agent after an edit if it skipped blast radius. Rejected: the harm has already occurred by the time the post-edit hook fires. Prevention is better than remediation.

### Block all edits until blast radius is checked

Block every Edit call regardless of symbol impact score. Rejected: creates constant friction for routine edits on small symbols. The per-session, high-impact-only design is intentional to keep the guard from becoming an annoyance that users disable on day two.

### Shared session state via external store

Persist the checked-symbol set across server restarts via SQLite or a file. Rejected for v0.8: in-process memory is sufficient and avoids a concurrency model. Revisit if multi-session continuity becomes a request.
